> **AX (AI-Augmented Experience)** — AI가 모든 것을 증강하는 시대
>
> 이 코스는 컴퓨터 과학, 소프트웨어 공학, 프로젝트 관리에 대한 사전 지식이 전혀 없는 사람을 위해 설계되었습니다.
> Agent Skills와 Duru-Conductor를 활용해서 AX 시대에 생각하고 행동하는 법을 배웁니다.

## 모듈 1: AX 시대란 무엇인가

### 세 가지 시대

![세 가지 시대 — 도구, 자동화, AX](images/01-three-eras.png)

**도구의 시대:** 망치, 계산기, 워드프로세서. 사람이 모든 결정을 내리고, 도구는 실행만 도움.

**자동화의 시대:** 공장 로봇, 스프레드시트 공식, 이메일 자동 발송. 반복적인 일은 기계가, 창의적인 일은 사람이.

**AX 시대 (지금):** AI가 전문가 수준의 판단을 할 수 있게 됨. 코드를 쓰고, 디자인을 하고, 전략을 짜고, 품질을 검사할 수 있음. **사람의 역할은 "실행"에서 "방향 설정"으로 바뀜.**

### 무엇이 달라졌는가

| 전 (Before AX) | 후 (After AX) |
|---|---|
| 홈페이지를 만들려면 개발자를 고용 (300-1000만원) | AI에게 설명하면 함께 만듦 (무료~최소 비용) |
| 사업계획서를 쓰려면 컨설턴트 경험 필요 | AI가 구조를 잡고 함께 작성 |
| 프로젝트 관리는 PM 교육과 도구 필요 | AI가 프로세스를 안내하고 추적 |
| 실수하면 복구가 어렵거나 불가능 | AI가 위험을 미리 경고하고 설명 |
| "기술력"이 진입 장벽 | "명확한 생각"이 핵심 역량 |

### 핵심 통찰

> **AX 시대에서 가장 가치 있는 능력은 코딩이 아닙니다.**
> **원하는 것을 명확하게 생각하고, 그것을 말로 표현하는 것입니다.**

이것은 단순한 격언이 아닙니다. Duru-Conductor의 접수 단계(7가지 차원)가 정확히 이 능력을 훈련시킵니다:

![구조적 사고의 7가지 차원](images/02-structured-thinking.png)

이 일곱 가지 질문은 소프트웨어에만 적용되는 것이 아닙니다. 식당 메뉴 리디자인, 동네 행사 기획, 학교 발표 자료에도 작동합니다. **구조적 사고(structured thinking)**라는 이 능력은 어디에나 전이됩니다.

## 모듈 2: 새로운 사고방식 — AX 시대의 7가지 전환

### 전환 1: "나는 못 해" → "나는 방향을 정하고, AI가 실행한다"

**옛날 생각:** "프로그래밍을 모르니까 앱을 못 만들어."
**새로운 생각:** "무엇을 만들고 싶은지는 아니까, AI에게 설명하면 함께 만들 수 있어."

이것은 게으름이 아닙니다. 건축가가 벽돌을 직접 쌓지 않는 것과 같습니다. 건축가의 가치는 **어떤 건물을 지을지 결정하는 것**이지, 벽돌을 쌓는 기술이 아닙니다. AX 시대에서 여러분은 건축가입니다.

**실습:** 지금 만들고 싶은 것 하나를 떠올려 보세요. 아무리 사소해도 좋습니다. "우리 동네 카페 메뉴를 예쁘게 정리한 웹페이지"도 완전히 괜찮습니다.

### 전환 2: "완벽하게 알아야 시작한다" → "시작하고 나서 배운다"

**옛날 생각:** "HTML, CSS, JavaScript를 다 배운 다음에 홈페이지를 만들어야지."
**새로운 생각:** "지금 당장 만들면서, 필요할 때 AI에게 물어보며 배우자."

Duru-Conductor의 네 단계가 이것을 자연스럽게 만듭니다. 접수 → 기획 → 실행 → 회고. 모든 것을 알 필요 없이, 각 단계에서 필요한 것만 알면 됩니다. 그리고 모르는 것은 AI가 채워줍니다.

**핵심:** 학습의 최고 형태는 "만들면서 배우는 것"입니다. 교과서를 100페이지 읽는 것보다 실제로 하나를 만드는 것이 10배 더 빨리 배웁니다.

### 전환 3: "대충 해도 된다" → "완전하게 하는 것이 거의 공짜다"

이것은 gstack의 Garry Tan이 **"Boil the Lake"(호수를 끓여라)**라고 부르는 철학에서 왔습니다.

**옛날 생각:** "테스트는 귀찮으니 대충 넘어가자." "에러 처리는 나중에."
**새로운 생각:** AI가 꼼꼼함의 비용을 거의 0으로 만들었으므로, **항상 완전한 버전을 만들자.**

Duru-Conductor의 노력 비교표가 이것을 숫자로 보여줍니다:

```
| 작업         | 사람 팀  | AI 활용 시 | 압축 비율 |
|-------------|---------|-----------|---------|
| 테스트 작성   | 2일     | 10분       | 288배   |
| 에러 처리     | 1일     | 5분        | 288배   |
| 문서 작성     | 3일     | 15분       | 288배   |
```

10분 투자로 2일치 작업을 할 수 있는데, 왜 빼겠습니까?

**단, 이것은 과잉 설계(over-engineering)와 다릅니다.** "호수"는 끓일 수 있지만, "바다"는 끓일 수 없습니다. 현재 프로젝트에 필요한 것을 완전히 하되, 미래에 필요할지도 모르는 것까지 만들지는 마세요.

### 전환 4: "혼자서 다 해야 한다" → "가상 팀을 조립한다"

**옛날 생각:** "풀스택 개발자가 아니면 앱을 못 만들어."
**새로운 생각:** "Duru-Conductor가 프로젝트에 맞는 가상 팀을 자동으로 조립해 준다."

Duru-Conductor의 12가지 역할:

| 분야 | 역할 | 하는 일 |
|---|---|---|
| 소프트웨어 | 설계자, 백엔드, 프런트엔드, 인프라, 연동, QA | 코드 작성 및 테스트 |
| 콘텐츠 | 연구원, 작가, 편집자, 비주얼 디자이너 | 글쓰기 및 디자인 |
| 비즈니스 | 기획자, 분석가, 전략가 | 분석 및 전략 수립 |

여러분은 이 역할들을 직접 수행하지 않습니다. AI가 각 역할의 관점에서 일합니다. 여러분은 **감독하고 결정하는 사람**입니다.

### 전환 5: "실수가 무섭다" → "안전망이 있으니 대담하게 시도한다"

**옛날 생각:** "터미널에서 뭔가 잘못 입력하면 다 날아갈 수 있으니 조심해야 해."
**새로운 생각:** "`/careful`이 위험을 미리 경고해 주니까 자신 있게 시도할 수 있어."

`/careful`의 철학은 **"설명하되 막지 않는다"**입니다. 위험한 명령이 감지되면:
1. 이 명령이 뭘 하는지 쉬운 말로 설명
2. 왜 위험한지 구체적으로 설명
3. 더 안전한 대안을 제시
4. 최종 결정은 여러분이

이것은 보호자가 아니라 **교사**입니다. 시간이 지나면 여러분이 스스로 위험을 인식하게 됩니다.

### 전환 6: "끝나면 끝이다" → "매번 성찰하고 성장한다"

**옛날 생각:** "프로젝트 끝! 다음으로 넘어가자."
**새로운 생각:** "잠깐, `/retro`로 이번에 뭘 배웠는지 정리하자. 다음에 더 잘할 수 있게."

회고는 선택이 아닙니다. 성장의 핵심 메커니즘입니다. Retro가 하는 것:

1. **사실 수집** — 실제 지표를 모아서 감정이 아닌 데이터로 판단
2. **구조적 성찰** — "잘된 것 / 어려웠던 것 / 다르게 할 것 / 배운 것"
3. **기록 저장** — RETROSPECTIVE.md로 남겨서 다음에 참조

**복리 효과:** 첫 프로젝트에서 3가지를 배우고, 두 번째에서 3가지를 더 배우고... 다섯 번째 프로젝트 때는 15가지의 교훈이 쌓여 있습니다. 이것이 경험입니다. 회고를 하지 않으면 같은 실수를 반복합니다.

### 전환 7: "기술이 가치다" → "판단력이 가치다"

**옛날 생각:** "React를 할 줄 아는 것이 나의 가치."
**새로운 생각:** "무엇을 만들어야 하는지, 왜 만들어야 하는지 판단하는 것이 나의 가치."

AI가 코드를 쓸 수 있게 되면서, "코드를 쓸 수 있다"는 것은 차별화가 되지 않습니다. 차별화는:
- **문제를 정확히 정의하는 능력** — Duru-Conductor 1단계(접수)가 훈련
- **좋은 판단을 내리는 능력** — Duru-Conductor 2단계(기획)가 훈련
- **품질을 평가하는 능력** — `/review`가 훈련
- **경험에서 배우는 능력** — `/retro`가 훈련

이것이 AX 시대의 진짜 능력입니다.

## 모듈 3: Agent Skills 이해하기

### 쉽게 말해서

**Agent Skill = AI에게 부여하는 전문성**

일반 AI에게 "홈페이지 만들어줘"라고 하면, 대략적인 답을 줍니다. 하지만 `/duru-conductor`를 활성화하고 같은 말을 하면, AI가:
- 7가지 차원으로 체계적으로 질문합니다
- 프로젝트에 맞는 팀을 자동으로 구성합니다
- 워크스트림 단위로 계획을 세웁니다
- 품질 게이트를 적용합니다
- 진행 상황을 파일로 기록합니다

같은 AI인데 **행동 방식이 완전히 달라집니다.** 이것이 스킬의 힘입니다.

### 스킬은 어떻게 생겼나요?

스킬의 정체는 **마크다운 파일** — 즉, 텍스트 파일입니다. 프로그래밍 언어가 아닙니다. 사람이 읽을 수 있는 지침서입니다. 이 지침서가 AI에게 "이런 상황에서는 이렇게 행동해라"라고 알려줍니다.

이것이 중요한 이유: **누구나 스킬을 읽고, 이해하고, 수정할 수 있습니다.** 프로그래밍 지식이 필요 없습니다.

### 스킬의 연결

Duru-Conductor의 스킬들은 **파일을 통해 서로 연결**됩니다:

![스킬 체이닝 — 파일을 통한 연결](images/03-skill-chaining.png)

이것은 인간 팀이 일하는 방식과 같습니다. 기획자가 기획서를 쓰면, QA가 기획서를 보고 테스트하고, 팀장이 결과를 보고 회고합니다. 문서가 사람들을 연결하듯, 아티팩트가 스킬들을 연결합니다.

## 모듈 4: 실습 — 첫 프로젝트

### Duru-Conductor의 네 단계 흐름

![Duru-Conductor의 네 단계 흐름](images/04-four-phases.png)

### 준비

1. AI 코딩 에이전트를 설치합니다 (Claude Code 추천)
2. Duru-Conductor를 설치합니다:
   ```bash
   npx skills add parandurume-labs/duru-skills
   ```
3. 만들고 싶은 것을 하나 정합니다

### 실습 과제: "우리 동네 빵집 홈페이지"

아래를 그대로 입력해 보세요:

```
/duru-conductor
우리 동네 빵집 홈페이지를 만들고 싶어요.
메뉴와 가격, 위치, 영업시간을 보여주고,
간단한 문의 양식이 있으면 좋겠어요.
무료로 만들고 싶고, 이번 주 안에 끝내고 싶어요.
```

### 따라가며 관찰할 것

Duru-Conductor가 진행하는 동안 이것들을 관찰하세요:

**1단계에서 관찰:**
- [ ] Duru-Conductor가 몇 가지 질문을 하나요? (5개 이하여야 합니다)
- [ ] 질문이 쉬운 말인가요? (전문 용어가 없어야 합니다)
- [ ] "모르겠어요"라고 하면 기본값을 제안하나요?
- [ ] 요약표를 보여주고 확인을 구하나요?
- [ ] SME(소상공인) 프로젝트를 감지했나요?

**2단계에서 관찰:**
- [ ] 어떤 팀을 구성했나요?
- [ ] 워크스트림이 합리적인 순서인가요?
- [ ] 노력 비교표를 보여주나요?
- [ ] `/careful`이나 `/review` 사용을 제안하나요?

**3단계에서 관찰:**
- [ ] 각 워크스트림 완료 후 알려주나요?
- [ ] BUILD-LOG.md를 생성하나요?
- [ ] 품질 게이트를 적용하나요?

**4단계에서 관찰:**
- [ ] RETROSPECTIVE.md를 만드나요?
- [ ] 잘된 점과 개선할 점 둘 다 포함하나요?

### 실습을 마치며

축하합니다! 첫 프로젝트를 완성했습니다. 이제 이 파일들이 프로젝트 폴더에 있을 것입니다:

```
내-빵집-홈페이지/
├── INTAKE.md          ← 합의한 내용
├── ARCHITECTURE.md    ← 기술 설계
├── BUILD-LOG.md       ← 진행 기록
├── RETROSPECTIVE.md   ← 배운 것
├── index.html         ← 실제 홈페이지
├── styles.css         ← 디자인
└── ...
```

## 모듈 5: 안전하게 일하기

### 왜 안전이 중요한가

컴퓨터 세계에는 "되돌리기"가 없는 작업이 있습니다:
- 파일 삭제 (휴지통이 없는 환경에서)
- 데이터베이스 내용 삭제
- git 히스토리 덮어쓰기
- 클라우드 리소스 삭제

초보자가 이런 것들을 실수로 실행하는 것은 충분히 일어날 수 있는 일입니다. 두려움이 아니라 **이해**로 대응해야 합니다.

### `/careful` 사용법

프로젝트 시작 시에 활성화하세요:

```
/careful
```

이제 AI가 위험한 명령을 실행하기 전에 항상 멈추고 설명합니다.

### 위험 등급 이해하기

![위험 등급 — CRITICAL, HIGH, MEDIUM](images/05-risk-levels.png)

| 등급 | 의미 |
|---|---|
| **CRITICAL** | 시스템 손상 또는 복구 불가능한 데이터 손실 |
| **HIGH** | 되돌리기 매우 어려움 |
| **MEDIUM** | 불편하지만 복구 가능 |

### 안전하게 일하는 습관

1. **삭제 전에 백업:** 파일을 삭제하기 전에 다른 곳에 복사하거나 이동
2. **git commit 자주 하기:** 작업을 자주 저장하면 되돌리기 쉬움
3. **"왜?"를 먼저 묻기:** 명령이 뭘 하는지 이해한 후에 실행
4. **`/careful` 항상 켜두기:** 초보 시절에는 항상 활성화

## 모듈 6: 품질을 확인하는 법

### 왜 검토가 중요한가

만든 것이 "작동한다"와 "좋다"는 다른 개념입니다:

| 작동만 하는 것 | 좋은 것 |
|---|---|
| 페이지가 뜬다 | 모바일에서도 잘 보인다 |
| 양식이 제출된다 | 스팸이 방지된다 |
| 데이터가 저장된다 | 개인정보가 보호된다 |
| 텍스트가 읽힌다 | 일관된 톤과 스타일이다 |

### `/review` 사용법

프로젝트가 완성된 후에 실행하세요:

```
/review
```

Review가 하는 일:
1. 프로젝트 유형을 자동 감지 (소프트웨어 / 콘텐츠 / 비즈니스)
2. 해당 유형에 맞는 체크리스트로 검사
3. 발견된 문제를 심각도별로 분류 (CRITICAL → HIGH → MEDIUM → LOW)
4. 구체적인 수정 방법 제안
5. `REVIEW.md`로 결과 저장

### 리뷰 결과 읽는 법

```
| # | 심각도     | 분류    | 발견 사항                | 파일        | 권장 조치 |
|---|----------|---------|------------------------|------------|---------|
| 1 | CRITICAL | 보안    | API 키가 코드에 하드코딩됨  | config.js  | 환경 변수로 이동 |
| 2 | HIGH     | 정확성   | 에러 처리가 누락됨        | api.js     | try-catch 추가 |
| 3 | MEDIUM   | 품질    | 코드 중복 발견           | utils.js   | 함수로 추출 |
```

**규칙: CRITICAL부터 먼저 고치세요.** HIGH, MEDIUM, LOW 순서로.

### 수정하기

Review 결과를 보고 "고쳐주세요"라고 하면 AI가 심각도 순서대로 수정합니다.

## 모듈 7: 성장하는 법 — 회고의 기술

### 왜 회고가 중요한가

같은 일을 10번 해도 회고 없이는 1번 한 것과 비슷한 수준에 머무릅니다. 회고를 하면:

![경험의 복리 효과](images/06-compound-growth.png)

이것이 **경험의 복리 효과**입니다. 회고 없는 10년 경력보다 회고 있는 1년이 더 성장합니다.

### `/retro` 사용법

프로젝트를 마친 후에 실행하세요:

```
/retro
```

Retro가 하는 일:
1. 있으면 Duru-Conductor 아티팩트를 자동으로 읽음
2. 없으면 직접 질문으로 맥락을 파악
3. 측정 가능한 지표를 자동 수집 (git 통계, 파일 수, 테스트 결과)
4. 구조적 성찰 질문을 안내
5. `RETROSPECTIVE.md`로 결과 저장

### 좋은 회고의 네 가지 질문

1. **잘된 것은?** — 구체적으로. "잘했다"가 아니라 "예약 시스템의 Google Sheets 연동이 예상보다 빨리 됐다"
2. **어려웠던 것은?** — 솔직하게. 어려웠던 것을 인정해야 다음에 대비할 수 있습니다
3. **다르게 할 것은?** — 행동으로 바꿀 수 있는 것으로. "더 열심히"가 아니라 "디자인을 먼저 확정한 후 코딩 시작"
4. **배운 것은?** — 가장 중요한 질문. 기술적인 것, 도메인에 대한 것, 자신의 작업 방식에 대한 것 모두 괜찮습니다

### 회고를 습관으로 만들기

- **매 프로젝트 후에** `/retro` 실행
- **짧아도 괜찮습니다** — 5분짜리 회고도 안 하는 것보다 100배 낫습니다
- **RETROSPECTIVE.md를 보관하세요** — 나중에 돌아보면 성장이 보입니다
- **패턴을 찾으세요** — 같은 문제가 반복되면, 프로세스를 바꿀 때입니다

## 모듈 8: AX 시대의 직업과 역할

### 사라지는 것과 남는 것

| 사라지고 있는 가치 | 남거나 커지는 가치 |
|---|---|
| 코드를 타이핑하는 속도 | 무엇을 만들지 판단하는 능력 |
| 특정 프레임워크 지식 | 문제를 정확히 정의하는 능력 |
| 반복적인 테스트 실행 | 품질 기준을 정하고 평가하는 능력 |
| 템플릿 문서 작성 | 사람의 필요를 이해하는 감수성 |
| 데이터 입력 | 데이터에서 의미를 찾는 해석력 |

### AX 시대의 가치 전환

![AX 시대의 가치 전환](images/07-value-shift.png)

### AX 시대에 가치 있는 7가지 역량

1. **구조적 사고** — 복잡한 것을 체계적으로 분해하는 능력
   - Duru-Conductor 1단계가 이것을 훈련시킵니다

2. **명확한 표현** — 원하는 것을 정확하게 말로 표현하는 능력
   - 좋은 프롬프트를 쓰는 것이 곧 좋은 커뮤니케이션입니다

3. **판단력** — 여러 선택지 중에서 적절한 것을 고르는 능력
   - Duru-Conductor 2단계에서 계획을 검토하고 승인하는 과정이 이것을 훈련시킵니다

4. **품질 감각** — "작동한다"와 "좋다"를 구분하는 능력
   - `/review`가 이것을 훈련시킵니다

5. **학습 능력** — 경험에서 의미를 추출하는 능력
   - `/retro`가 이것을 훈련시킵니다

6. **도메인 지식** — 특정 분야에 대한 깊은 이해
   - AI는 범용적이지만, "우리 동네 미용실 고객이 뭘 원하는지"는 여러분만 압니다

7. **공감 능력** — 사용자/고객/독자의 필요를 이해하는 능력
   - AI가 대체할 수 없는 가장 인간적인 능력입니다

### 실천 가이드

**첫째, 만들어 보세요.**
이론이 아니라 실천입니다. `/duru-conductor`로 프로젝트 하나를 시작하세요. 완벽할 필요 없습니다.

**둘째, 안전망을 쓰세요.**
`/careful`을 켜고 대담하게 시도하세요. 실수가 교사입니다.

**셋째, 검토하세요.**
`/review`로 자기 작업을 객관적으로 봅니다. 불편하더라도 직면하세요.

**넷째, 성찰하세요.**
`/retro`를 빠뜨리지 마세요. 이것이 성장의 복리 효과를 만듭니다.

**다섯째, 반복하세요.**
두 번째 프로젝트, 세 번째 프로젝트... 매번 더 나아집니다. 이것이 AX 시대에 전문가가 되는 방법입니다.
